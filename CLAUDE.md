# STM32F407 Telemetry Gateway - AI Agent Instructions

## Project Overview

This is the **main firmware for a fault-tolerant telemetry gateway** for ROV/submarine robotics. The STM32F407 Discovery board acts as the central coordinator, managing dual-radio communication (Wi-Fi via ESP32 + LoRa fallback), link quality monitoring, and black-box SD card logging.

**Current Status**: Phase 2 in progress вҖ” SX1278 LoRa driver integration. Phases 3-5 (FreeRTOS tasks, state machine, logging) pending.

**Key Architecture**: 
- Primary link: ESP32 (Wi-Fi, USART2) вҶ’ UDP to ground station
- Fallback link: SX1278 (LoRa, SPI1) вҶ’ 433 MHz to ground station  
- Monitoring: Real-time link quality metrics (RSSI, packet loss, latency)
- Storage: MicroSD card via SDIO for post-mission analysis

## Hardware & Pin Configuration

### Microcontroller
- **MCU**: STM32F407VG Discovery board (Cortex-M4, 168 MHz, 1 MB Flash, 192 KB SRAM)
- **Clock**: HSE 8 MHz crystal вҶ’ PLL вҶ’ 168 MHz

### Communication Interfaces

#### USART2 (ESP32 Wi-Fi Gateway)
- **STM32 Pins**: PA2 (TX) / PA3 (RX)
- **Speed**: 115200 baud (hardcoded in ESP32 firmware)
- **Protocol**: Binary telemetry packets (DEPTH, HDG, BAT, TEMP fields)
- **Note**: RX connected to ESP32's RX2 on GPIO16 (D5 on WeMos D1 R32)

#### SPI1 (LoRa SX1278 Radio)
- **Clock**: PA5 (SCK) at ~10 MHz
- **MOSI**: PA7 вҶ’ SX1278 DIN
- **MISO**: PA6 вҶҗ SX1278 DOUT
- **Chip Select (NSS)**: PC4 (GPIO OUT, active LOW)
- **Reset**: PC5 (GPIO OUT, active LOW)
- **Interrupt (DIO0)**: PB0 (GPIO IN, triggers TX/RX done)
- **Note**: PA5/PA6/PA7 shared with onboard MEMS accelerometer; PE3 kept HIGH to deselect MEMS during LoRa ops

#### MicroSD Card (SDIO)
- **Interface**: SDIO peripheral (1-bit or 4-bit mode, TBD in Phase 5)
- **Status**: Not yet integrated; FatFS driver pending

## Code Organization & Conventions

### Main Application Entry (Core/Src/main.c)

The `main()` function follows STM32CubeMX standard layout:
1. **System Init**: `MX_*_Init()` calls for GPIO, UART, SPI, clock
2. **User Setup**: USER CODE section for application state initialization
3. **Infinite Loop**: HAL polling or interrupt-driven handling (FreeRTOS scheduler when available)

**Current Phase 1-2 Style**: Polling-based with blocking calls. Will transition to FreeRTOS in Phase 3.

### Driver Integration Pattern

Example: SX1278 LoRa driver (Phase 2 work)
```
Drivers/
в”ңв”Җв”Җ STM32F4xx_HAL_Driver/  вҶҗ ST vendor HAL (untouched)
в”ңв”Җв”Җ SX1278/                вҶҗ External LoRa driver
в”Ӯ   в”ңв”Җв”Җ SX1278.h
в”Ӯ   в”ңв”Җв”Җ SX1278.c
в”Ӯ   в”ңв”Җв”Җ SX1278_hw.h        вҶҗ Hardware abstraction layer
в”Ӯ   в””в”Җв”Җ SX1278_hw.c        вҶҗ STM32-specific SPI + GPIO glue
```

**Key Convention**: External drivers (like SX1278) receive a `_hw_t` structure passed in, which contains HAL handles (`SPI_HandleTypeDef`, GPIO ports, pins). This decouples the driver from STM32-specific code.

### Interrupt Handlers (Core/Src/stm32f4xx_it.c)

- Auto-generated stubs by STM32CubeMX
- Add user code in `/* USER CODE BEGIN IRQHandler_Name */` sections
- Common handlers: `USART2_IRQHandler`, `SPI1_IRQHandler`, `EXTI0_IRQHandler` (LoRa DIO0 interrupt)

## Development Workflow

### When Modifying Pin Configuration or Clocks

1. **Open telemetry_gateway.ioc** in STM32CubeMX
2. Make pin/clock/peripheral changes
3. **Generate Code** (Ctrl+S) вҖ” this overwrites `Core/Inc/main.h`, `stm32f4xx_hal_conf.h`, etc.
4. **Preserve User Code**: STM32CubeMX will not overwrite USER CODE sections
5. Re-build: `make clean && make`

**Danger Zone**: Never hand-edit files outside USER CODE blocks in `Core/Inc/Src/`; CubeMX regeneration will overwrite your changes.

### Adding a New Peripheral

Example: Adding SDIO for SD card in Phase 5:
1. Open `.ioc` in CubeMX
2. Enable SDIO peripheral, configure pins (PD0-3, etc.) and clock
3. Generate code
4. Create `Drivers/FatFS/` driver wrapper
5. In `main.c`, add USER CODE to initialize FatFS and mount SD card
6. Test mount and file operations before integrating with logging state machine

### Debugging with Cortex-Debug

- Requires ST-Link USB connected to STM32 board
- VS Code: Debug вҶ’ "Cortex Debug" launch config
- Breakpoints, watches, registers, memory inspector available
- **Note**: Debugging halts the processor; Wi-Fi/LoRa timings may suffer if breakpoint held for >100ms

## Critical Development Notes

### вҡ пёҸ SPI1 & MEMS Conflict
- PA5, PA6, PA7 are shared between SX1278 LoRa (SPI1) and onboard accelerometer (also SPI1)
- **Mitigation**: Keep PE3 GPIO output HIGH to hold accelerometer /CS asserted (chip disabled)
- This is already handled in `stm32f4xx_hal_msp.c`; do NOT reconfigure without understanding impact

### вҡ пёҸ UART2 Baud Rate
- ESP32 firmware hardcoded to 115200 baud
- If you change this via CubeMX, you **must** also update ESP32 firmware
- Sync them or you'll lose all telemetry data

### рҹ”ҙ Phase 2 Current Work: SX1278 Integration
- Driver source: [wdomski/SX1278](https://github.com/wdomski/SX1278) (already in repo)
- Next steps:
  1. Verify SPI1 DMA initialization in `MX_SPI1_Init()`
  2. Implement interrupt handler for LoRa DIO0 (TX/RX done)
  3. Add test routine to transmit/receive LoRa packets
  4. Verify RSSI and packet loss detection

### вҸӯпёҸ Phase 3 Transition: FreeRTOS
- Currently all logic is in `main()` loop (blocking)
- Phase 3 will split into tasks:
  - `telemetry_task` вҖ” read UART2, parse packets
  - `wifi_task` вҖ” manage ESP32 connection status
  - `lora_task` вҖ” LoRa TX/RX, DIO0 interrupt handling
  - `link_monitor_task` вҖ” RSSI sampling, state transitions
  - `logger_task` вҖ” SD card writes (Phase 5)
- Kernel: FreeRTOS v10.x (will be added to Drivers/)

## Recommended Agent Behavior

When working on this codebase:

1. **Before building**: Verify `.ioc` and Makefile are in sync (timestamps)
2. **Before editing Core/Src/*.c**: Check for `/* USER CODE BEGIN/END */` markers
3. **After CubeMX regeneration**: Run `make clean && make` to catch new/removed symbols
4. **On SPI or UART changes**: Double-check corresponding hardware pin config in `stm32f4xx_hal_msp.c`
5. **On interrupt work**: Review `stm32f4xx_it.c` for existing handlers; add user code only in marked sections
6. **Before suggesting refactors**: Understand Phase 2 вҶ’ Phase 3 FreeRTOS migration will restructure code heavily; focus on immediate blocker fixes
7. **On flashing issues**: Verify ST-Link connection and OpenOCD config before suggesting code changes
